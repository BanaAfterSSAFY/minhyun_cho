# 🚀 빛의 속도로 거듭제곱 계산하기: 모듈러 거듭제곱 완전 정복 (백준 1629번)

안녕하세요, 알고리즘과 자료구조의 세계를 탐험하는 여러분! 오늘은 백준 온라인 저지 1629번, "곱셈" 문제를 통해 거대한 숫자의 거듭제곱을 효율적으로 계산하는 마법 같은 기법, **모듈러 거듭제곱(Modular Exponentiation)** 또는 **이진 거듭제곱(Binary Exponentiation)**에 대해 깊이 파헤쳐 볼 시간입니다. 이 기법은 분할 정복(Divide and Conquer) 패러다임의 아름다움을 잘 보여주는 대표적인 예시죠!

## 1. 문제 요약 및 풀이 전략

### 📖 문제 요약

주어진 문제는 세 개의 자연수 A, B, C를 입력받아 `A를 B번 곱한 결과를 C로 나눈 나머지`를 출력하는 것입니다. 즉, `(A^B) % C`를 계산해야 합니다.
여기서 주의할 점은 A, B, C가 각각 최대 2,147,483,647 (Integer.MAX_VALUE)에 달할 수 있다는 점입니다. 이는 단순히 A를 B번 반복해서 곱하다가는 `long` 타입의 범위를 넘어설 뿐만 아니라, `B`가 매우 클 경우 시간 제한 초과(Time Limit Exceeded)가 발생할 것이라는 강력한 힌트입니다.

### 💡 풀이 전략: 분할 정복을 이용한 모듈러 거듭제곱

일반적인 `A^B` 계산은 `B`번의 곱셈을 필요로 합니다. 하지만 우리는 `B`가 지수라는 점과 모듈러 연산의 특징을 활용하여 곱셈 횟수를 `log B` 수준으로 줄일 수 있습니다.

핵심 아이디어는 다음과 같습니다:

1.  **모듈러 연산의 분배 법칙**: `(X * Y) % C == ((X % C) * (Y % C)) % C`
    *   이 성질 덕분에 우리는 중간 계산 결과가 아무리 커지더라도, 각 곱셈 연산 후 `C`로 나눈 나머지를 취함으로써 오버플로우를 방지하고 숫자를 작게 유지할 수 있습니다.
2.  **지수 법칙의 활용**:
    *   **B가 짝수일 때**: `A^B = A^(B/2) * A^(B/2)`
    *   **B가 홀수일 때**: `A^B = A^(B/2) * A^(B/2) * A`
    *   이 법칙을 재귀적으로 적용하면, `B`를 매번 절반으로 줄여나갈 수 있습니다.

예를 들어, `A^10`을 계산한다면:
`A^10 = (A^5) * (A^5)`
`A^5 = (A^2) * (A^2) * A`
`A^2 = (A^1) * (A^1)`
`A^1 = A` (기저 사례)

이 과정을 재귀 함수로 구현하여 `B`가 `0`이 될 때까지 반복합니다. `B`가 0일 때는 `A^0 = 1`이므로 1을 반환하도록 합니다.

```java
// 핵심 재귀 함수 로직
private static long divide(long a, long b) {
    // 1. 기저 사례: b가 0이면 a^0 = 1
    if (b == 0) {
        return 1 % C; // C=1일 때 0을 반환하기 위해 1 % C로 처리
    }

    // 2. 재귀 호출: a^(b/2) 계산
    long temp = divide(a, b / 2);

    // 3. 결과 계산: (a^(b/2) * a^(b/2)) % C
    long res = (temp * temp) % C;

    // 4. b가 홀수이면, a를 한 번 더 곱해줌
    if (b % 2 == 1) {
        res = (res * a) % C;
    }

    // 5. 최종 결과 반환 (한 번 더 % C를 해주어도 무방, 이미 res가 C보다 작으므로)
    return res % C;
}
```

## 2. 시간 복잡도와 공간 복잡도 분석

### ⏱️ 시간 복잡도 (Time Complexity)

*   `divide` 함수는 지수 `B`를 `B -> B/2 -> B/4 -> ... -> 1` 과 같이 절반씩 줄여가며 재귀 호출합니다.
*   이 과정은 `log₂B` 단계에 걸쳐 진행됩니다.
*   각 단계(재귀 호출)에서는 상수 시간의 연산(곱셈, 나머지 연산)이 이루어집니다.
*   따라서 전체 시간 복잡도는 **O(log B)** 입니다.
*   `B`가 `2 * 10^9` 정도라면, `log₂(2 * 10^9)`는 대략 `31` 정도밖에 되지 않아 매우 효율적입니다.

### 💾 공간 복잡도 (Space Complexity)

*   재귀 함수 호출은 스택 메모리를 사용합니다.
*   재귀의 깊이는 지수 `B`가 `1`이 될 때까지 `log₂B`번 호출되므로, 최대 `log₂B`만큼의 스택 프레임이 쌓입니다.
*   각 스택 프레임은 소수의 지역 변수를 저장하므로 상수 공간을 차지합니다.
*   따라서 전체 공간 복잡도는 **O(log B)** 입니다.

## 3. 코드에서 개선할 점이나 주의해야 할 예외 케이스 (Edge Case)

제시된 코드는 모듈러 거듭제곱의 표준적인 구현으로, 대부분의 경우 잘 동작합니다. 하지만 몇 가지 세부 사항을 짚어볼 수 있습니다.

### 1. 전역 변수 `C`의 활용 (개선할 점)

```java
static long C; // C가 전역 변수로 선언되어 있음
// ...
C = Integer.parseInt(st.nextToken()); // main에서 할당
// ...
private static long divide(long a, long b) {
    // ... C를 직접 사용
}
```

*   **현재 코드**: `C`가 `static` 전역 변수로 선언되어 `divide` 함수에서 직접 접근합니다.
*   **고려 사항**: 전역 변수를 사용하는 것은 코드를 간결하게 만들 수 있지만, 몇 가지 단점이 있습니다.
    *   **재사용성 감소**: `divide` 함수가 다른 `C` 값으로 호출될 수 있는 유틸리티 함수로 확장될 경우, 전역 `C`를 바꾸거나 다른 방법으로 처리해야 합니다.
    *   **캡슐화 위반**: `divide` 함수의 외부 상태(`C`)에 의존하게 되어 독립성이 떨어집니다.
    *   **멀티스레드 환경**: 만약 이 코드가 멀티스레드 환경에서 여러 `C` 값으로 동시에 사용된다면 문제가 발생할 수 있습니다.
*   **개선 방안**: `C`를 `divide` 함수의 매개변수로 전달하는 것이 일반적으로 더 좋은 프로그래밍 습관입니다.

    ```java
    // 변경 후
    public static void main(String[] args) throws IOException {
        // ...
        long C_val = Integer.parseInt(st.nextToken()); // C를 지역 변수로 선언
        System.out.println(divide(A % C_val, B, C_val)); // C를 인자로 전달
    }

    private static long divide(long a, long b, long C_mod) { // C_mod로 받음
        if (b == 0) {
            return 1 % C_mod; // C_mod 사용
        }
        long temp = divide(a, b / 2, C_mod); // C_mod 전달
        long res = (temp * temp) % C_mod;
        if (b % 2 == 1) res  = (res * a) % C_mod;
        return res % C_mod;
    }
    ```
    이 방식은 함수가 필요한 모든 정보를 인자로 받아들이기 때문에 더 독립적이고 재사용하기 좋습니다.

### 2. 초기 `A % C` 처리 (매우 중요)

```java
System.out.println(divide(A % C, B));
```

*   `main` 함수에서 `A`를 바로 `C`로 나눈 나머지를 `divide` 함수에 넘기는 것은 아주 훌륭하고 중요한 처리입니다.
*   **이유**: `(A^B) % C == ((A % C)^B) % C` 라는 모듈러 연산의 성질 때문에, 초기 `A` 값이 `C`보다 훨씬 크더라도 미리 `A % C`로 줄여 놓으면 이후의 모든 곱셈 연산에서 오버플로우를 방지하고 계산 효율을 높일 수 있습니다. 만약 이 처리를 하지 않았다면, `A`가 매우 큰 값일 때 `A * A`만 해도 `long` 타입을 초과할 수 있습니다.

### 3. 기저 사례 `b == 0` 처리 (예외 케이스)

```java
if (b == 0) {
    return 1 % C;
}
```

*   `A^0 = 1`이므로, `B`가 `0`일 때 `1`을 반환하는 것은 맞습니다.
*   `1 % C`로 처리한 부분은 `C`가 `1`인 경우를 정확하게 처리합니다.
    *   만약 `C = 1`이라면 `A^B % 1`은 항상 `0`이 되어야 합니다. `1 % 1`은 `0`을 반환하므로, 이 코드는 `C = 1`인 경우에도 올바르게 `0`을 반환합니다. 만약 `return 1;`이었다면 `C = 1`일 때 오답이 됩니다. 따라서 현재의 `return 1 % C;`는 완벽한 처리입니다.

### 4. `long` 타입 사용 (예외 케이스)

*   입력 `A, B, C`는 `int` 범위까지 주어질 수 있지만, 중간 계산 결과인 `temp * temp`나 `res * a`는 `long` 범위를 넘어설 수 있습니다.
*   코드에서는 모든 관련 변수(`A`, `B`, `C`, `temp`, `res`)를 `long` 타입으로 선언하여 오버플로우 가능성을 최소화하고 있습니다. 이는 올바른 접근입니다.
*   `C`가 `int` 범위지만, `A % C`의 결과가 `long`으로 타입 캐스팅되고, 중간 계산이 `long`으로 진행되는 것이 안전합니다.

### 5. 음수 처리 (문제에서는 자연수로 명시)

*   이 문제에서는 A, B, C가 자연수로 주어지므로 음수에 대한 걱정은 없습니다.
*   만약 음수 지수나 음수 밑을 허용하는 문제였다면, 추가적인 처리가 필요했을 것입니다. (예: `A^-B = 1 / (A^B)`)
